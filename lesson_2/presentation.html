Haskell is Call-By-Need!

  • Values aren't immediately evaluated
  • A pointer to a "thunk" that can compute the value is stored instead
  • When the value becomes needed, this "thunk" is ran
  • The evaluated value replaces the thunk so you don't evaluate twice

What's a list?

                emptyList = Empty
                oneList = Cons 10 Empty
                twoList = Cons 10 (Cons 11 Empty)

  • Lists are either empty or an item and the rest of the list

So where is Haskell lazy?

  • Haskell evaluates things as shallowly as possible
  • Starts off completely unevaluated
  • If you pattern match on a part of it, that part is evaluated, by the rest is not!
  • Doing side effects under this is tricky—who knows when they'll actually happen?
  • Monadic IO is essentially required for sanity as it enforces a sequence

What's a fold?

  • Take an initial value
  • Take a function that takes the initial value and the next value in the list
  • Merge them together
                
                    foldr (+) 0 [1, 2, 3]
                    1 + (foldr (+) 0 [2, 3])
                    1 + (2 + (foldr (+) 0 [3]))
                    1 + (2 + (3 + (foldr (+) 0 [])))
                    1 + (2 + (3 + 0))
  • The associativity matters
  • Left folds are left-associative, so they aren't as lazy!
  • Generally you want to use foldl', which isn't lazy at all
                
                    foldl (+) 0 [1, 2, 3]
                    foldl (+) (0 + 1) [2, 3]
                    foldl (+) ((0 + 1) + 2) [3]
                    foldl (+) (((0 + 1) + 2) + 3) []
                    (((0 + 1) + 2) + 3)

FUNCTOR ALERT

What's a functor?

  • A typeclass (or a set of functions you can implement for any type, with laws)
  • Consists of one required function, fmap
  • Allows you to change values with a context
              
                  class Functor f where
                    fmap :: (a -> b) -> f a -> f b

                  instance Functor Maybe where
                    fmap f (Nothing) = Nothing
                    fmap f (Just a) = Just (f a)

Why does this matter?

  • Functors are very general, and apply to many data structures
  • They let you write highly generic code
  • Saves quite a bit of time
  • A good introduction to polymorphism
  • Notice: the contained type can change!

Combining things in a context

What's an Applicative?

  • Also a typeclass
  • A sub-class of functor
  • Allows you to combine items with a context
  • Allows you to put items into a context
  • Allows you to run a function over two arguments in a context
              
                  class Appliactive f where
                    pure :: a -> f a
                    <*> :: f (a -> b) -> f a -> f b
                    liftA2 :: (a -> b -> c) -> (f a -> f b -> f c)

pure in depth

  • We have some value
  • For some reason, we need the value in a context
  • Maybe we need it as a Maybe
  • Maybe we need it in a list
              
                  pure :: a -> Maybe a
                  pure a = Just a

liftA2 in depth

  • We have a function that takes two arguments
  • We have two arguments in a context
  • We want to run the function over those arguments
  • Basically fmap with two arguments!
              
                  liftA2 :: (a -> b -> c) -> Maybe a -> Maybe b -> Maybe c
                  liftA2 _ Nothing _ = Nothing
                  liftA2 _ _ Nothing = Nothing
                  liftA2 f (Just a) (Just b) = Just $ f a b
